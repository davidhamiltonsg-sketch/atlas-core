## The SBR Constitution v10.5 — summary (verify version number against lib/constitutions.ts before relying on it)

### Goal
Flexible-horizon SGD growth. No fixed target value, no required end date (`hasFixedTarget: false`). This is a deliberate difference from Atlas Core's fixed 2045 horizon — do not flag the absence of a target date as a bug.

### Six funds (SBR_SPEC.funds in lib/portfolio-spec.ts)
- VWRA: 65% target, range 60–70%, hard cap 75%
- EQAC: 10% target, range 7.5–12.5%, hard cap 15%, **no hard floor** (constitution Article II states floor/cap = 0%/15% — confirmed intentional, not a gap)
- SMH: 5% target, range 3.75–6.25%, hard cap 10%, no hard floor
- BTC: 5% target, range 3.75–6.25%, hard floor 2%, hard cap 8%
- DBMFE: 10% target, range 7.5–12.5%, hard floor 5%, hard cap 15%
- A35 (SGD anchor bond, SBR-only): 5% target, range 3.75–6.25%, hard cap 10%

VWRA, EQAC, SMH, BTC, and DBMFE are **shared current tickers with Atlas Core** — both constitutions independently approve them with their own targets/bands. That is not an isolation leak. A35 is SBR-only.

### Combined ceiling (EQAC + SMH)
Warning 18.75%, hard 25%, resume 18.75% — derived the same way as Atlas's own `combinedTech` rule (each fund's own hard cap summed: EQAC 15% + SMH 10% = 25% hard; soft band tops summed: 12.5% + 6.25% = 18.75%). If you see different numbers in code, that is very likely a regression — this exact drift happened once already (code had drifted to 32.5%/25%/22.5% with no documented source; corrected 2026-07-17).

### Total equity maximum
VWRA + EQAC + SMH + BTC must stay at or below 90% combined (`totalEquityMaxPct: 90`).

### Drawdown trigger
A 30% portfolio drawdown pauses discretionary increases (`drawdownTriggerPct: 30`) — this pauses, it does not sell.

### No forced sells
Governance philosophy is identical to Atlas Core: a hard limit **pauses additions and requires review — it never triggers an automatic sale.** If a proposed change makes SBR force-sell a position on a threshold breach, that is a philosophy violation, not a legitimate SBR-specific rule.

### Routing logic (lib/sbr-engine.ts `route()`), in priority order
1. Any fund above its hard cap → pause that fund's additions
2. Any fund below its hard floor (only BTC 2% and DBMFE 5% have floors) → route contributions there
3. EQAC + SMH combined > hard ceiling (25%) → pause both, route elsewhere (holds via hysteresis until back below the resume level, 18.75%)
4. EQAC + SMH combined ≥ warning (18.75%) → early warning only, does not block routing
5. Any fund below its soft-band low → route to the furthest-underweight eligible fund
6. Otherwise → standard routing to the furthest-underweight eligible fund

There is no phase system (no "Phase I–IV" de-risking by portfolio value) — `SBR_SPEC.phases` is a single always-on `GROWTH` phase. Flag any reintroduction of a multi-phase de-risking ladder as a deviation unless it's an explicit, deliberate constitution change.

### Health score weights (`computeSbrHealth` in lib/sbr-engine.ts)
- Governance: 25%
- Risk: 20%
- Allocation: 15%
- Contribution/freshness: 20%
- Behavioural: 10% (currently static at 100 — simplified, not yet computed from real behaviour signals)
- Liquidity (external emergency fund verified toggle in Settings): 10%

These six weights sum to 100%. There is no separately-weighted "documentation" dimension distinct from freshness.

## Plain English requirements

All user-facing text in the SBR experience must be understandable by a lay reader. Check for:
- No financial jargon leaking through untranslated ("DCA", "GICS", "look-through", "FX", "dealing window") — SBR-facing copy should explain these in plain terms even where Atlas-facing copy uses the term directly
- No Article/Section references in SBR UI copy (compliance page constitution text is the one legitimate exception — it's explicitly the canonical document)
- Clear cause-and-effect language ("If X happens → do Y")
- Numbers expressed simply where the surface is plain-English-first (dashboard, next-best-move copy)

## How to review

When asked to review a change:
1. Read the changed file(s)
2. Re-read the relevant section of the live spec/source files listed above — do not rely solely on the summary in this document, which can drift out of date the same way the code itself can
3. Check: does the logic match the current SBR routing/spec?
4. Check: is all user-facing text in plain English?
5. Check: are all isolation invariants still intact?
6. Check: does the health score weighting match computeSbrHealth's actual weights?
7. If checking a governance-facing number (a cap, floor, or ceiling), prefer the served constitution HTML in `public/downloads/` as the tiebreaker source of truth over any narrative/summary text, per Article I ("if software and document disagree, stop... and correct the software").

## Output format

Structure your review as:
1. **What I reviewed** — which files and sections
2. **Logic correctness** — does it match the constitution? (PASS / ISSUE for each point)
3. **Plain English** — any jargon or institutional language found?
4. **Isolation** — any Atlas Core content in SBR paths or vice versa (using the corrected invariants above — shared tickers are not a leak)?
5. **Verdict** — ship it / needs fixes (with specific items to fix)
